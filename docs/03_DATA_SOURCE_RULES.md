# 03_DATA_SOURCE_RULES - 데이터 수집 규칙

## 키움 API Provider 사용 규칙

키움증권 API는 자동매매가 아니라 분석 데이터 Provider로만 사용한다.

주문 전송, 시장가 주문, 조건주문, 자동매매 기능을 구현하지 않는다.

키움 Provider는 현재가, 체결, 거래량, 거래대금, 분봉, 일봉 데이터를 표준 OHLCV로 변환한다.

표준 컬럼:

```text
DateTime 또는 Date
Open
High
Low
Close
Volume
TradeValue
```

키움 현재가 수집 실패, 체결 데이터 부족, 분봉 생성 실패, 대표가격 산정 불가, 키움 현재가와 일봉 대표 가격의 비정상 불일치, 필수 지표 계산 실패, 장중/장외 상태 판정 실패는 정상 보고서 저장 금지 대상이다.

키움 데이터 검증 실패 시 최종판정은 `분석 중단`이며 QA 실패 파일만 저장한다.

키움 일봉은 대표 일봉 후보로 사용할 수 있으나 단독 사용은 금지한다.

키움 일봉과 pykrx/FDR 최신 완료 일봉의 종가 차이가 1%를 초과하면 경고, 3%를 초과하면 분석 중단으로 처리한다.

키움 현재가의 `prev_close`가 있으면 pykrx/FDR 최신 완료 종가와 비교한다.

- 차이 1% 초과: 경고
- 차이 3% 초과: 분석 중단

키움 현재가가 0 이하이거나 timestamp가 없으면 분석 중단한다.

키움 장중 고가/저가가 있으면 현재가가 해당 범위 안에 있어야 한다.

## 국내 주식 데이터 우선순위

1. 네이버금융 현재가 및 일별시세
2. pykrx
3. FinanceDataReader
4. yfinance `.KS` 또는 `.KQ`
5. KRX 공개 데이터
6. 다음금융 또는 기타 공개 페이지

## 미국 주식 데이터 우선순위

1. yfinance
2. Yahoo Finance 공개 페이지
3. Nasdaq 공개 페이지
4. Stooq
5. MarketWatch 공개 페이지
6. 기타 공개 데이터

## 네이버 사용 규칙

국내 장중 현재가의 1차 소스로 사용한다.

가능하면 아래 항목을 수집한다.

- 현재가
- 전일 종가
- 당일 시가
- 당일 고가
- 당일 저가
- 장중 누적 거래량
- 조회 시각

## pykrx / FinanceDataReader 사용 규칙

완료 일봉 교차검증과 국내 장중 미확정 일봉 보조 확인에 사용한다.

데이터가 비어 있거나 지연되면 실패 사유를 남긴다.

## 데이터 불일치 유형

데이터 검증 실패 또는 경고는 아래 유형으로 분리한다.

- 동일 거래일 가격 불일치: 같은 최신거래일인데 OHLC 가격 차이가 허용 범위를 넘는 경우
- 동일 거래일 거래량 불일치: 같은 최신거래일인데 거래량 차이가 허용 범위를 넘는 경우
- 최신거래일 불일치: 둘 이상의 주요 소스가 서로 다른 최신거래일을 제시하는 경우
- 보조 소스 최신거래일 지연: pykrx/FDR 등 대표 소스는 일치하지만 yfinance 같은 보조 소스만 하루 이상 늦는 경우
- 핵심 소스 수집 실패: pykrx 또는 FinanceDataReader 등 대표 가격 판단에 필요한 소스가 실패한 경우
- 수급 데이터 부족: 가격 검증과 분리하며 수급 판단만 보류하는 경우

국내 주식에서 pykrx와 FinanceDataReader가 같은 최신거래일과 같은 OHLCV를 제시하면 대표 가격으로 사용한다.

yfinance가 pykrx/FDR보다 최신거래일이 늦으면 대표 가격 산정에서 제외하고 데이터검증 섹션에 `보조 소스 최신거래일 지연`으로 표시한다.

이 경우 가격 데이터 전체 불일치로 처리하지 않는다.

## 대표 가격 선정과 정밀 판단 중단

국내 주식 대표 가격 우선순위:

1. pykrx와 FinanceDataReader가 최신거래일·OHLCV 기준으로 일치하면 해당 값을 대표 가격으로 사용
2. 핵심 소스가 하나만 가능하고 보조 소스가 같은 최신거래일·가격으로 일치하면 보수적으로 사용
3. yfinance만 stale이면 yfinance는 제외하고 pykrx/FDR 대표 가격을 사용
4. 대표 가격을 산정할 수 없으면 정밀 판단 중단

정밀 판단 중단 조건:

- pykrx/FDR/yfinance 중 2개 이상에서 최신거래일 또는 가격이 충돌
- 대표 가격을 산정할 수 없음
- 동일 거래일 가격 차이가 1.0%를 초과
- 핵심 소스가 모두 실패

pykrx/FDR이 일치하고 yfinance만 stale인 경우에는 정밀 판단 중단이 아니라 `보조 소스 지연으로 보수 판단`으로 표시한다.

신뢰도 산정:

- pykrx/FDR 일치 + yfinance stale: 가격 신뢰도 중간~높음, 거래량 신뢰도 중간~높음, 교차검증 완전성 중간
- 수급 데이터 부족: 가격 신뢰도를 낮추지 않고 수급 신뢰도만 낮음/데이터 부족으로 표시

## yfinance 사용 규칙

국내 종목의 yfinance suffix:

| 시장 | suffix |
|---|---|
| KOSPI | `.KS` |
| KOSDAQ | `.KQ` |

yfinance는 보조 확인 소스다.

국내 분봉 데이터가 실패해도 네이버/pykrx/FDR 장중 데이터가 정상이라면 전체 장중 현재가 신뢰도를 무조건 낮음으로 내리지 않는다.

분봉 데이터 신뢰도만 별도로 표시한다.

## yfinance 분봉 실패 처리

yfinance 1분봉 수집 실패 또는 데이터 없음이면 아래처럼 표시한다.

```text
yfinance 분봉: 수집 실패 또는 데이터 없음
분봉 차트: 생성 가능 시 보조 참고만 적용
분봉 데이터 신뢰도: 부분 통과 또는 경고
```

실패 상태에서 `yfinance 분봉: 정상`이라고 쓰지 않는다.

## NXT/통합 거래량 분리

NXT/통합 참고 거래량은 정규장 거래량과 분리한다.

보고서에는 아래를 구분한다.

- 정규장 거래량
- NXT/통합 참고 거래량
- 장중 누적 거래량
- 20일 평균 거래량 대비

## 시장 지수 심볼 규칙

국내 시장 지수는 아래를 사용한다.

| 시장 | 주 비교 지수 |
|---|---|
| KOSPI | KOSPI |
| KOSDAQ | KOSDAQ |

허용 심볼:

```text
KS11
KQ11
1001
2001
```

KOSPI 정상 범위는 1,000~6,000으로 본다.

KOSDAQ 정상 범위는 300~2,000으로 본다.

비정상 범위 또는 업종지수 혼입 시 QA 실패다.

## 수급 데이터 fallback

수급 데이터는 아래 순서로 재시도한다.

1. 네이버 투자자별 매매동향 페이지
2. pykrx 투자자별 순매수 데이터
3. FinanceDataReader 또는 KRX 공개 데이터
4. 실패 시 수급 판단 보류

수급 실패 시 아래 표현을 쓰지 않는다.

- 외국인/기관 동반 순매수
- 외국인/기관 동반 순매도
- 수급 우호
- 수급 악화
- 매도 흐름 강화

반드시 “수급 데이터 부족으로 수급 판단 보류”라고 쓴다.

## 섹터 라벨 규칙

보고서의 실패 조건, 시장/섹터 환경, 프로 트레이더 판단, 최종 판단에 들어가는 섹터 문구는 종목 코드와 종목명으로 판별한다.

기본 매핑:

| 코드 | 섹터 라벨 |
|---|---|
| 033100 | 전력기기/변압기 |
| 267260 | 전력기기 |
| 010120 | 전력기기/자동화 |
| 298040 | 전력기기/중공업 |
| 403870 | 반도체 장비/공정 장비 |
| 005930 | 반도체/IT 대형주 |
| 000660 | 반도체 메모리 |
| 010140 | 조선/중공업 |
| 034020 | 원전/에너지 |
| 196170 | 바이오 |
| 247540 | 2차전지 소재 |

매핑이 없으면 종목명을 보고 추론한다.

판별 실패 시 `동종 업종`을 사용한다.

관련주/동종업종 데이터가 부족하면 특정 섹터 흐름을 단정하지 않는다.

섹터 문구는 보고서 핵심 판단에 필요한 경우에만 짧게 쓴다.

종목별 매핑이 없거나 확신이 없으면 특정 산업명을 만들지 말고 `동종 업종`이라고 쓴다.

수급, 섹터, 관련주 데이터가 실패하면 가격/거래량/지표 중심의 제한 판단으로 표시한다.
